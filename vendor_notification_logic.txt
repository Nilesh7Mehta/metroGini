2. HOW NOTIFICATIONS ARE FETCHED (Vendor API)
----------------------------------------------

Route  : GET /api/vendor/notifications
Auth   : Bearer token (vendor)
Scoped : Only notifications belonging to the vendor's own user_id

Query params:
  category = all | orders | rider   (maps to reference_type filter)
  page     = 1, 2, 3...
  limit    = max 50, default 20

category mapping:
  all    → no filter (returns everything)
  orders → reference_type = 'order'  (order received + deadline reminders)
  rider  → reference_type = 'rider'  (delivery completed by rider)


3. OTHER ENDPOINTS
-------------------

GET  /api/vendor/notifications/unread-count   → returns { unread_count: N }
PATCH /api/vendor/notifications/read-all      → marks all as read
PATCH /api/vendor/notifications/:id/read      → marks one notification as read


4. NOTIFICATION TABLE COLUMNS USED
------------------------------------

  user_id        → vendor's user_id (from vendors table)
  title          → notification heading
  message        → detail text
  reference_type → 'order' or 'rider'
  reference_id   → order id
  is_read        → false by default, updated via PATCH endpoints
  created_at     → timestamp
